  ---
  2. 🔴 多供应商降级链（Fallback Chain）

  来源：LangChain Router、a-stock-data 数据源优先级策略

  实践：单供应商不可用时自动切换，不中断服务：

  主选（DeepSeek）→ 备选（Qwen）→ 本地兜底（Ollama）

  项目现状：目前 create_client() 只根据配置创建单一客户端，无降级链机制。

  最佳实践做法：

  pub struct LlmRouter {
      primary: Box<dyn LlmClient>,
      fallback: Option<Box<dyn LlmClient>>,
      local: Option<Box<dyn LlmClient>>,  // Ollama 兜底
  }

  impl LlmRouter {
      pub async fn chat_with_fallback(&self, messages: &[ChatMessage], config:
  &LlmConfig) -> Result<LlmResponse> {
          // 1. 尝试主选
          match self.primary.chat(messages, config).await {
              Ok(resp) => return Ok(resp),
              Err(e) => tracing::warn!("主选失败: {}, 尝试备选", e),
          }
          // 2. 尝试备选
          if let Some(fallback) = &self.fallback {
              match fallback.chat(messages, &fallback_config).await {
                  Ok(resp) => return Ok(resp),
                  Err(e) => tracing::warn!("备选失败: {}, 尝试本地", e),
              }
          }
          // 3. 本地兜底
          self.local.chat(messages, &local_config).await
      }
  }

  ---
  3. 🔴 结构化输出降级（Structured Output Degradation）

  来源：TradingAgents structured.py + 项目已有的 supports_structured_output()

  实践：国产模型对 response_format: {"type": "json_object"}
  的支持程度参差不齐，必须有三级降级：

  结构化输出成功 → 自由文本 + JSON 解析 → 原始文本兜底

  项目现状：✅ 已部分实现。openai_compat.rs 有 supports_structured_output() 和
  supports_structured_output_for_provider()，engine.rs:409 有
  serde_json::from_str 解析降级。但缺少 主动关闭 Thinking 模式再请求结构化输出
  的逻辑。

  关键坑点：

  ┌───────────┬──────────────────┬───────────┬────────────────────────────┐
  │           │                  │    非     │                            │
  │  供应商   │ Thinking 模式下  │ Thinking  │          处理方式          │
  │           │                  │   模式    │                            │
  ├───────────┼──────────────────┼───────────┼────────────────────────────┤
  │ DeepSeek  │    ❌ 不支持     │  ✅ 支持  │ 维度分析时关闭             │
  │ v4-flash  │ response_format  │           │ Thinking，综合决策时开启   │
  ├───────────┼──────────────────┼───────────┼────────────────────────────┤
  │ DeepSeek  │    ❌ 不支持     │  ✅ 支持  │ 同上                       │
  │ v4-pro    │ response_format  │           │                            │
  ├───────────┼──────────────────┼───────────┼────────────────────────────┤
  │ Qwen3.7   │  ✅ 两者都支持   │  ✅ 支持  │ 无需特殊处理               │
  ├───────────┼──────────────────┼───────────┼────────────────────────────┤
  │ GLM-5     │  ✅ 两者都支持   │  ✅ 支持  │ 无需特殊处理               │
  └───────────┴──────────────────┴───────────┴────────────────────────────┘

  最佳实践做法：

  // 维度分析：需要 JSON 输出 → 关闭 Thinking
  if self.needs_structured_output && provider == CloudProvider::DeepSeek {
      body["thinking"] = serde_json::json!({"type": "disabled"});
      body["response_format"] = serde_json::json!({"type": "json_object"});
  }

  // 综合决策：需要深度推理 → 开启 Thinking，不要求 JSON
  if self.needs_deep_reasoning {
      // 不设 response_format，让模型自由输出
      body["thinking"] = serde_json::json!({"type": "enabled", "budget_tokens":
  10000});
  }

  5. 🟡 Token 预算控制与成本透明

  来源：astock-peg analysis.ts、TradingAgents StatsCallbackHandler

  实践：

  1. 输入截断：原始数据可能超 token 限制 → 截断到预算内
  2. 成本预估：分析前告知用户大概花多少钱
  3. 成本统计：分析后展示实际花费

  项目现状：✅ schemas.rs 有 TokenBudget，engine.rs 有 CostEstimator（在 S19
  规格中定义但代码未实现）。

  国产模型特别注意：

  // GLM 中文 token 比率 ~1:1.6（1 token ≈ 1.6 个汉字）
  // DeepSeek/Qwen 中文 token 比率 ~1:1.5
  // 这意味着同样的中文内容，GLM 消耗的 token 数约为英文的 60-70%
  // 成本预估时需要按中文 token 比率校准
  pub fn adjust_for_chinese(provider: CloudProvider, char_count: usize) -> u32 {
      let ratio = match provider {
          CloudProvider::GLM => 1.6,
          _ => 1.5,
      };
      (char_count as f64 / ratio) as u32
  }

  // 3. 相同股票的维度分析可复用前几轮的缓存
